# Deepen スキル — ヒアリングで論点を深堀り

issue または論点を対話的に深堀りし、**決定を出す**ところまでが責務。
記録(ADR・作業ログ)は `/record` に委譲する。

---

## フロー

### 1. 論点を把握する

引数が issue 番号(`#NN`)の場合、`mcp__github__issue_read` で内容を取得する。

```
owner: "masa-dev-2000"
repo: "kyoryokutai_support"
issue_number: <番号>
```

取得した内容 + ユーザーの補足から以下を整理:
- **問い**: 何を決めようとしているか(1行)
- **背景**: なぜ今決める必要があるか
- **選択肢**: A / B / C(2〜4択、自分で洗い出す)

### 2. AskUserQuestion でヒアリング(2〜3ターン)

```
質問設計の原則:
- 行動ベース: 「どっちが好き?」→「どういうときに使う?」
- 技術用語を使わない
- 大→小の順に絞る
- 1ターンに2〜3問、選択肢は4つ以内
```

### 3. 決定をまとめる

ヒアリング結果から:
- **決定内容**(1〜3行で断言)
- **理由**(ヒアリングの根拠を紐付け)
- **影響範囲**(削除・変更・追加される要素)

ユーザーに決定内容を提示し、合意を確認する。

### 4. GitHub Issue を更新する

`mcp__github__issue_write` で issue にヒアリング結果と決定を追記する:

```
method: "update"
owner: "masa-dev-2000"
repo: "kyoryokutai_support"
issue_number: <番号>
state: "closed"  # 決定が出た場合
body: <元の本文 + ## ヒアリング結果 + ## 決定>
```

### 5. /record を促す

決定が出たら:
> 「決定をADRと作業ログに記録しますか? `/record` で起票できます。」

記録するかどうかはユーザーが判断する。

---

## 原則

- **探索フェーズ**: 決定を出すことが目的、記録は `/record` の仕事
- ヒアリングは **2〜3ターン** で完結させる。長引かせない
- 「まだ決まらなかった」場合は issue を open のまま残し、途中経過をコメントする
