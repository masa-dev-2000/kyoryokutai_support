# 設計決定スキル(ヒアリング → ADR + Issue)

設計議論をユーザーにヒアリングしながら進め、決定事項を **ADR**(`docs/19_v5_adr.md`) と **GitHub Issue** に同時記録する。

---

## フロー

### 1. 論点を整理する

ユーザーの入力から以下を抽出:
- **問い**: 何を決めようとしているか(1行)
- **背景**: なぜ今決める必要があるか
- **選択肢**: A / B / C(2〜4択)

選択肢が不明確な場合はまず自分で洗い出す。

### 2. AskUserQuestion でヒアリング

`AskUserQuestion` を使って1ターンに2〜3問まで質問する。  
選択肢は **4つ以内**、説明は **1行**。

```
質問の設計原則:
- ユースケース・行動パターンから聞く(「どっちが好き?」ではなく「どういうときに使う?」)
- 技術用語は使わない(FAB ではなく「右下の丸い+ボタン」)
- 段階的に絞る(大→小の順)
```

ヒアリングは **2〜3ターン** で完結させる。長引かせない。

### 3. 決定をまとめる

ヒアリング結果から:
- **決定内容**(1〜3行で断言)
- **理由**(ヒアリングの根拠を紐付け)
- **影響範囲**(削除・変更・追加される要素)

### 4. ADR に追記する

`docs/19_v5_adr.md` の末尾に追記する。

```markdown
## ADR-XXX: <タイトル>

**確定日:** YYYY-MM-DD

### Context
<なぜ決める必要があったか>

### Decision
<何を決めたか(箇条書きで明確に)>

### Consequence
<何が変わるか / 何をやらないか>

### ヒアリング根拠
| 質問 | 回答 |
|---|---|
| ... | ... |
```

ADR番号は既存の最大番号+1。

### 5. GitHub Issue に記録する

`mcp__github__issue_write` で issue を作成する:
- **owner**: `masa-dev-2000`
- **repo**: `kyoryokutai_support`
- **title**: `[ADR-XXX] <タイトル>`
- **body**: ADR と同じ内容 + 実装タスクのチェックリスト

---

## ADR フォーマット参照

既存ADRは `docs/19_v5_adr.md` にある。  
ADR番号は末尾のものを確認してから採番する。

## GitHub Issue 参照

`mcp__github__issue_write` の使い方は `/github-issue` スキルを参照。

---

## 使用例

```
ユーザー: /design-decision 日報タブを廃止して月報に統合するか
→ 1. 論点整理(日報タブの役割・月報との重複を分析)
→ 2. ヒアリング(利用シーン・入力動線・遡り単位)
→ 3. 決定まとめ(廃止 or 統合 or 残す)
→ 4. ADR-XXX として docs/19_v5_adr.md に追記
→ 5. GitHub Issue を作成
```
