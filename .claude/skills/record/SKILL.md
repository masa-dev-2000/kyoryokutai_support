# Record スキル — ADR + 作業ログ記録

`/deepen` で出た決定や、完了した作業を永続化する。

---

## モード選択

ユーザーの入力から記録の種類を判断する:

| 入力の性質 | モード |
|---|---|
| 設計・アーキテクチャの決定 | **ADR モード** |
| 作業・実装の完了記録 | **作業ログ モード** |
| 両方 | 両方実行 |

---

## ADR モード

### 1. 内容を確認する

`/deepen` の結果、または会話から以下を抽出:
- 決定内容
- 理由
- 影響範囲
- ヒアリング根拠(あれば)

### 2. `docs/19_v5_adr.md` に追記する

既存の最大ADR番号を確認してから採番(末尾を Read して確認)。

```markdown
## ADR-XXX: <タイトル>

**確定日:** YYYY-MM-DD

### Context
<なぜ決める必要があったか>

### Decision
<何を決めたか(箇条書きで明確に)>

### Consequence
<何が変わるか / 何をやらないか>

### ヒアリング根拠
| 質問 | 回答 |
|---|---|
| ... | ... |
```

### 3. GitHub Issue を作成する

`mcp__github__issue_write` で:
- **title**: `[ADR-XXX] <タイトル>`
- **body**: ADR本文 + 実装タスクのチェックリスト
- **labels**: `design`

```
method: "create"
owner: "masa-dev-2000"
repo: "kyoryokutai_support"
```

---

## 作業ログ モード

### 1. 内容を確認する

会話から以下を抽出:
- 何をしたか(完了タスク)
- 変更したファイル・機能
- 次のアクション(あれば)

### 2. `docs/20_work_log.md` に追記する

ファイルが存在しない場合は新規作成。

```markdown
## YYYY-MM-DD

### 完了
- <完了タスク>

### 変更ファイル
- <ファイルパス>: <変更内容>

### 次のアクション
- <todo>(あれば)
```

### 3. 関連 issue をクローズする(任意)

完了に伴いクローズすべき issue があれば `mcp__github__issue_write` で state: "closed" に更新する。

---

## 原則

- ADR は「決定した事実」だけを書く。検討中の内容は書かない
- 作業ログは箇条書きで簡潔に。詳細は commit message と issue に任せる
- `/record` はいつでも呼べる。`/deepen` の直後でなくてもよい
