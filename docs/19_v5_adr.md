# ADR v5 ─ 設計判断記録

v5「シンプル版・業務ツール調」で確定した設計判断のまとめ。
各 ADR は **Context → Decision → Consequence** の三部構成。

## ADR-001: 役場側を「承認 / 月報 / お知らせ」の 3 タブに固定する

**確定日:** 2026-06-11

### Context
- 役場担当は副業で協力隊を見ているケースが多く、業務工数を増やせない
- 戦略 v3.2 では役場側 3 機能を「状況確認 / 情報共有 / 申請承認」と定義していたが、
  状況確認に KPI ダッシュボード / 議会報告 / ベンチマークまで含むなど範囲が広すぎた

### Decision
**画面 = 仕様** として、v5 試作で固まった以下の 3 タブを真とする:
1. **承認**(経費 / 月次報告 / 活動相談を統合)
2. **月報**(担当隊員グリッド ・ 状態一覧)
3. **お知らせ**(textarea + 一斉送信デフォルト)

設定はフッタリンクに隠す(初回・異動時のみ)。

### Consequence
- 役場担当の操作対象が「承認 1 タップ + 月 1 回の月報チェック + 必要時のお知らせ」に圧縮された
- 議会報告書 PDF / 全国事例閲覧 / KPI ダッシュボード / ベンチマーク / 県横断ダッシュボードを
  **MVP 外に降格**(ADR-002 参照)
- ドキュメント側を画面に寄せる(`docs/11_feature_inventory.md` v3 として記録)

## ADR-002: 議会報告書 PDF・全国事例閲覧(役場)・県横断 DB を MVP 外に降格

**確定日:** 2026-06-11

### Context
- 戦略 v3.2 で「議会報告書 PDF」は役場側の最大価値とされていた
- 県庁スポンサー向けには「県横断ダッシュボード」も価値
- どちらも v5 試作で UI 露出すると画面が複雑化、ADR-001 の方針と衝突

### Decision
以下を MVP 外に降格し、Year 2 で再評価する:
- 議会報告書 PDF 自動生成(役場側)
- 全国事例閲覧 UI(役場側)
- KPI / 進捗ダッシュボード(役場側)
- 全国・近隣自治体ベンチマーク
- 県横断ダッシュボード(県庁モード)
- 承認時の常時コメント必須(ADR-003 参照)

### Consequence
- 県庁提案資料では「**次フェーズの拡張余地**」として明示
- 全国事例は隊員側に集中(`docs/11_feature_inventory.md` で Tier 2.5 に格下げ)
- 県庁価値は Year 2 で別レイヤー(`/v5/prefecture` 等)として復活させる

## ADR-003: 役場の承認コメントは「差戻し時のみ必須」

**確定日:** 2026-06-11

### Context
- 戦略 v3.2 では「**承認コメント必須化**」が豊岡 kintone の弱点逆張りとされていた
- 一方で毎回コメント必須化は役場側の負担を増やし、ADR-001 と衝突
- 「承認はワンタップ」「差戻しのみ理由を求める」が現実的なバランス

### Decision
- 承認:1 タップ即決(コメント任意)
- 差戻し(「確認」ボタン):**5 文字以上のコメント必須**

### Consequence
- 双方向コミュニケーションの趣旨は「差戻し時の理由共有」に絞られた
- 役場側の心理的ハードルが下がる
- 承認自体に AI の verdict 色を残すと「AI が判定してる」見え方になるので、
  色分けは撤廃(ADR-004 参照)

## ADR-004: AI verdict の色分けを撤廃、「AI は判定しません」を全画面で統一

**確定日:** 2026-06-11

### Context
- 隊員側メンターでは「※ AI は判定しません。視点と材料のみ提供します。」を明記
- 一方、役場側承認カードの初期実装で verdict が緑「承認」/ 琥珀「確認」と色分け
- 隊員側と役場側で原則が分裂しており、戦略の整合性が崩れていた

### Decision
- 役場側の verdict カラー(緑 / 琥珀)を **slate 系に統一**
- 承認・差戻しボタンは色ではなく**文言とアイコンの違い**で区別
- 詳細シートに「※ AI は判定しません。視点と材料のみ提供します。」を再掲

### Consequence
- AI = 第三者として「材料を提供する」ポジションが全画面で一致
- 役場側の承認判断は人間に責任が残る(行政の責任構造との整合)
- 色を増やしていないので、視覚的な情報密度も上がらない

## ADR-005: 「日報」を「活動報告」に改名、活動分類を 2 軸に分離

**確定日:** 2026-06-12

### Context
- 「日報」は 1 日 1 件の前提を持つ語感
- 現実には 1 日複数件の活動を別々に記録したいケースが多い
- 「カテゴリ」を 1 軸で持つと、隊員ごとの特化テーマ(空き家担当 / 商店街活性化担当 等)に
  対応できず、全員に空き家が出るような不自然さが生まれる

### Decision
- 「日報」→「**活動報告**」に改名(1 日 N 件可能)
- カテゴリを 2 軸に分離:
  - **活動の種類**(共通固定):会議 / 出張 / 現場訪問 / 広報 / 内勤 / イベント / 振り返り / その他
  - **活動内容**(ユーザー編集可):空き家 / 移住相談 / 町報 / ... ─ 隊員のテーマに合わせて追加・削除
- 活動時間を必須入力(0.5h 単位)
- メモ欄に 5W1H ヒントを表示

### Consequence
- 役場側の「**月報**」も同じ用語感で一貫(活動報告 → 月報)
- 月報詳細でカレンダー表示できる素材が揃う(時間・経費・件数)
- 隊員ごとに活動内容ラベルが違うため、月報生成 AI のプロンプトを
  実態に合わせやすくなる

## ADR-006: 経費を「申請(事前)」「精算(事後)」のサブタブに分離

**確定日:** 2026-06-12

### Context
- 初期実装では「経費」タブの中で申請フォームに「領収書を撮る」ボタンがあった
- だが申請(事前承認)時点では領収書はまだ存在しない
- ユーザー指摘:申請と精算は別フェーズで、領収書は精算側にあるべき

### Decision
- 経費タブを「**申請**」「**精算**」のサブタブに分離
- **申請シート**(事前):タイトル / 金額 / 用途内容 のみ(領収書なし)
- **精算シート**(事後):実支出額 / 領収書添付(必須)/ 精算メモ
- 詳細シートで「未精算」状態を出し、「精算する」CTA で精算シートへ遷移

### Consequence
- 役場側の承認モードと対応が取れる:
  - 隊員「申請」→ 役場「**活動相談 / 経費(事前)**」承認
  - 隊員「精算」→ 役場「**経費(精算)**」承認
- 用語上の整理:**活動相談 = 事前承認、経費 = 精算承認**(画面で明示)

## ADR-007: AI 相談を「目的別」に分散する

**確定日:** 2026-06-12

### Context
- ヘッダー右上の「相談」ボタンは全機能対応で、抽象的だった
- 抽象的な対話より、用途を絞ったほうが AI の応答精度が上がる
- 「経費の用途を書く」「月報の来月計画を書く」「活動メモを整える」「事例を探す」など、
  目的別に必要な視点・参考情報が違う

### Decision
- 入力ごとに**インライン相談ボタン**を配置(4 箇所):
  - 活動報告のメモ欄(`daily-write`)
  - 月報の来月計画欄(`report-plan`)
  - 経費の用途欄(`expense-purpose`)
  - 事例の検索欄(`case-find`)
- 各シートは目的別の intro / hint / mock 応答を持つ(`consultMeta`)
- 応答に「**反映して閉じる**」ボタンを置き、元入力欄に AI 提案を adopt できる
- ヘッダーの「相談」は**相談メニュー**に変更(4 目的のエントリ)

### Consequence
- AI に渡るコンテキストが明示化される(`ConsultContext` 型 union)
- 全般相談を撤廃したことで、抽象的な質問のループが起きない
- 将来 LLM の system prompt は目的別に分離して設計できる

## ADR-008: Sheet をスタック管理化(`Sheet[]`)

**確定日:** 2026-06-12

### Context
- 初期実装は `sheet: Sheet | null` で 1 階層のみ
- 月報詳細(カレンダー)→ 日付タップ → 日詳細 → 「閉じる」で**月報一覧まで戻ってしまう**
- ユーザー期待:「閉じる」で 1 階層前(=カレンダー)に戻りたい

### Decision
- `sheets: Sheet[]` のスタック構造に変更
- `pushSheet(s)` / `popSheet()` / `closeAllSheets()` の API
- スタック深度 > 1 のとき、SheetHeader の「閉じる」を「**カレンダーに戻る**」等の戻りラベルに

### Consequence
- 複数階層モーダル遷移が直感的になる
- 相談シート → メニュー → 目的別シートの遷移も同じ仕組みで動く
- 将来モバイル ネイティブ実装に移しやすい構造(モーダルスタックは iOS/Android で標準)

## ADR-009: ハブ画面を業務ツール調に統一、管理者モードを追加

**確定日:** 2026-06-12

### Context
- v5 入口(/v5)は当初「Game-style UI」のグラデ + 絵文字キャラ
- 中の業務ツール調(白基調 + slate)と完全に浮いていた
- ユーザー指摘:「v5 にあってなさすぎ」
- かつ、組織管理(隊員台帳 / 職員 / 担当割当)を扱う面が役場側 3 タブには収まらず、
  別ロールが必要

### Decision
- ハブ画面を業務ツール調に統一(白基調 + slate)
- 「プレイヤー / マスター」→「**隊員 / 役場 / 管理者**」の 3 経路に
- 機能名を実装どおりに明示(活動報告 / 月報 / 経費 / 事例 等)
- トップページ(/)の v5 ラベルも「シンプル版・業務ツール調」に変更
- 管理者画面(/v5/admin)を新規追加:
  - 隊員台帳(隊員の追加・編集・退任)
  - 職員(役場側ユーザー管理)
  - 担当割当(職員 × 隊員 のマッピング)

### Consequence
- 入口から中身まで一貫した質感(白基調 + slate + 検索エンジン型)
- 役場側の「設定」(管轄隊員チェックリスト)と管理者側の「担当割当」が役割分離
  - 役場:自分の管轄を確認・調整
  - 管理者:組織全体の割当ルールを編集
- 管轄が変わる時は管理者が更新 → 役場側に自動で反映される構造

## ADR-010: レイアウトのメリハリは色ではなく「構図・余白・太さ」で

**確定日:** 2026-06-12

### Context
- 活動報告画面で「目が滑る」「どこを見るべきか分からない」ユーザー指摘
- カラフルにすればメリハリが付くが、業務ツール調 + 「白基調 + slate」の方針と衝突

### Decision
- 色を増やさず、以下の手段でメリハリを作る:
  - **太い区切り線**(`border-b-2 border-slate-900`)で主役セクションを区切る
  - 数字を 28px 黒太字(`text-[28px] font-black`)で目立たせる
  - サブセクションは細い線(`border-b border-slate-200`)+ 小ラベルで控えめに
  - カードを撤廃しリスト中心の構造に
- 経費の使用状況サマリ(2026-06-12 修正):
  - 旧:カード 3 つ + プログレスバー
  - 新:カードなし、5 行の薄いセクションに集約

### Consequence
- 視線が「主役 → サブ」の縦動線で流れる
- 色を増やさないので情報密度は上がるが視覚ノイズは増えない
- 業務ツールとしての一貫性が保たれる

## ADR-011: 全国事例の匿名化方針 ─ 隊員は実名 opt-in、関係者は自動匿名化

**確定日:** 2026-06-12

### Context
- ADR-002 / 設計初版では全国事例を「完全匿名化」する方針だった
- だが完全匿名化には次の問題がある:
  - 誰がやったか分からないため、参考にしたい隊員が**役場に問い合わせ**してしまい役場負担が増える
  - 「あの隊員に会いに行く」「直接コンタクトする」という Moat ─ つまり
    **隊員間の横ネットワーク形成**という最大の価値が失われる
- 一方で、活動関係者(住民・小売店等)の個人情報は保護する必要がある

### Decision
- **隊員本人:opt-in で実名公開**
  - 初期は **OFF**(隊員の心理ハードルを下げ、安全側に振る)
  - 隊員がプロフィール設定で ON にすると、以降の公開事例に実名 + 自治体名が出る
  - 個別プロジェクト単位で例外設定も可能
- **自治体名:常に公開**
  - 「あの町の取り組み」として他自治体が参考にできる
  - 受け入れ地域の評価にもつながる
- **関係者の匿名化:Claude で自動処理**
  - 個人名 → 「A さん」「住民の方」「協力者」等
  - 民間企業名 → 「地元の小売店」「町内の建設会社」等(規模・業種は残す)
  - **公的団体(自治会・観光協会・学校 等)は実名で残す**(文脈の価値を保つ)
- **連絡導線:PoC では実装しない**
  - 事例画面に隊員名と自治体名のみ表示
  - コンタクトは隊員側が SNS / 名刺等の既存手段で
  - Year 1 後半でシステム内連絡フォーム(メアド非公開、スパム防止)を追加

### Consequence
- 「隊員 = 公的活動者」「関係者 = 保護対象」の線引きが明確になる
- 隊員間ネットワーク形成と関係者保護の両立
- 役場の問い合わせ負担を抑制(誰がやったか分かるので直接行ける)
- 匿名化処理に Claude を 1 回追加で挟む(コストとレイテンシは数秒オーダー、許容範囲)
- 公的団体と民間企業の判別精度が課題 → プロンプトで例示 + 隊員レビュー画面で
  最終確認 → 公開、の二段構えにする(PoC では隊員確認のみで OK)
- 連絡フォームを Year 2 に持ち越したため、PoC 期間中はネットワーク効果が
  弱まる可能性があるが、まず「事例の質」を確保することを優先

## ADR-012: 経費承認は多段階ルートを隊員ごとに定義する

**確定日:** 2026-06-13

### Context
- ヒアリングで判明:承認は **隊員ごと・経費種別ごと** に経路が異なる
  - シンプル:企画課のみ(1 段)
  - 中:担当課(商工観光等)→ 企画課(2 段)
  - 複雑:担当課 → 受入団体 → 企画課(3 段)
- v3 までは「役場担当 1 人が承認」を前提にしており、現場と齟齬
- 受入団体が **財布を握っている** ケースが多く、団体長の承認なしには動かない

### Decision
- **`approval_routes`** に隊員 × 経費種別ごとのルートを定義
- **`approval_route_steps`** に各ステップを順序付きで持つ
  - `step_no` / `approver_type`(`dept` / `host_org` / `admin`)/ `approver_id`(具体ユーザー or NULL でロール解決)
- 申請発生時に `approval_route_steps` を順に展開して `approvals` レコード群を生成
- `approvals.step_no` で現在ステップを示す
- 上位ステップが pending な限り下位は不可視
- 既定ルートは自治体単位で 3 種類(シンプル / 中 / 複雑)を提供、隊員プロフィールで選択
- 受入団体ユーザーは `users.role = manager` の派生型として扱い、`organization_type = host_org` で識別

### Consequence
- データモデルがやや複雑化(承認テーブルが 2 個増)
- ただし運用パターンの差を吸収する唯一現実的な構造
- UI では経費・月報・活動相談の各承認画面に **タイムライン表示**(現在ステップ可視化)
- 受入団体を扱うため `host_organizations` テーブルを Identity サブシステムに追加
- 既定 3 種ルートで PoC は十分カバー、特殊運用は Year 2 で隊員個別カスタム化

## ADR-013: お知らせ・ルール・通知は右上ベル + ピン留めで統合する

**確定日:** 2026-06-13

### Context
- 隊員側にお知らせ閲覧画面が存在しなかった(v5 試作の欠落)
- ヒアリング:**ルール / Q&A** を常時参照したいニーズあり(自治体や受入団体が定めた経費ルール等)
- ルールがお知らせのタイムラインで流れて見えなくなることを避けたい

### Decision
- **右上ベル UI** を全画面に常駐(サイドメニューに「お知らせ」項目は追加しない)
- ベル押下で **お知らせ詳細パネル**(ドロワー)が右からスライドイン
- パネル内構成:
  - 上段:**ピン留めセクション**(`is_pinned = true` のお知らせ)
  - 下段:時系列リスト(新着順)
- お知らせには **kind** を持たせる:`info`(通常)/ `rule`(ルール)/ `qa`(Q&A)
- ルール / Q&A は基本的にピン留めで運用、自治体管理者が投稿
- **経費作成画面**には独立した「ルール参照ボタン」を **ヘッダーに常設**
  - ボタン押下でサイドパネルが開き、`announcements.kind in ('rule', 'qa')` を絞り込み表示
  - 経費判定の段階 1(MVP)に該当する機能

### Consequence
- お知らせ画面を独立ページとして作る必要がない(ドロワー UI のみ)
- ルール表示は **お知らせデータの再利用**(別テーブルにしない)= データ重複ゼロ
- 経費作成中に「これ通るかな?」と思った瞬間に同一画面で確認できる動線
- ルール本文を RAG 化するのは段階 2(Year 1 後半)、PoC は表示のみ
- `announcements.kind` と `is_pinned` の追加が必要

## ADR-014: 経費申請の動線は二系統 + 親子構造で表現する

**確定日:** 2026-06-13

### Context
- 経費の入力タイミングは 3 パターン:
  - **少額・日常的**(ボールペン、印刷費):活動中の現場で発生、レシート即添付
  - **中額・単発**(備品、印刷物):オフィスで計画的に申請
  - **複数日にまたがる**(出張、研修):事前に見積もり申請 → 帰着後にレシート別精算
- ヒアリングで「日報でレシートを出す」実態が確認されたため、日報経由動線は必須
- 旅費等は **見積もりが先に発生し、精算は複数レシートに分かれる**

### Decision
**動線を二系統で残し、旅費等は親子構造で表現する。**

#### 動線 ①:日報経由(現場で発生する少額経費)
- 日報入力時に「経費明細」セクションで `[+ 経費を追加]`(複数追加可能)
- 各明細は `タイトル(任意)` / `金額` / `用途` / `レシート画像` を持ち、保存時に `expenses` レコードとして自動生成
- `expenses.source_activity_log_id` + `source_receipt_index` で識別
- 申請前は日報からも経費画面からも編集可能、申請後は **経費画面が真実の源**

#### 動線 ②:経費画面の直接作成(中額・複数日経費)
- 経費画面 → 「新規作成」ボタン
- 単発経費:従来通り 1 件作成
- **出張等の見積もり経費:** `expense_kind = trip_parent` で親レコード作成
  - 親には `amount_requested`(見積もり)、`period_start` / `period_end`(出張期間)を持たせる
  - 親の詳細画面で `[+ レシート追加]` で子レコード(`expense_kind = trip_receipt`、`parent_expense_id` で紐付け)
  - 親の `amount_settled` は子の合計値を自動集計(計算カラム or トリガー)

#### 二重申請防止
- `expenses` に **部分 UNIQUE INDEX**:`(source_activity_log_id, source_receipt_index) WHERE source_activity_log_id IS NOT NULL`
- 日報の経費明細を再保存しても、`source_receipt_index` が同じなら UPDATE、新規分のみ INSERT
- 申請後にスイッチを OFF にしても経費レコードは削除されず、`status = 取下げ` に遷移(監査ログに残す)

#### 編集ルール
- **申請前**(`status = 下書き`):日報・経費画面どちらからも編集可
- **申請後**(`status = 申請中` 以降):経費画面のみで編集可、日報側は表示のみ(灰色化)
- 差戻し後の再申請:経費画面で修正 → 再提出(日報は触らない)
- 隊員が日報の金額を変更したい場合:申請を取下げ → 編集 → 再申請

#### UI
- **日報画面**:活動内容の下に「💴 経費」セクション、明細リスト + `[+ 経費を追加]`
- **経費画面リスト**:`📝 日報から` / `🚄 出張(見積もり)` / `📄 レシート(出張の子)` のバッジで識別
- **親経費の詳細画面**:子レシート一覧 + 進捗バー(`amount_settled / amount_requested`)
- **新規ボタン**:`[ 単発 ]` / `[ 出張(見積もり) ]` の 2 つに分岐

### Consequence
- 動線が 2 つあるため UI 説明が増えるが、現場実態に沿った設計
- 親子構造で旅費の「見積もり vs 実支出」の差分管理が自然に表現できる
- 部分 UNIQUE で二重申請を DB レベルで防げる
- 月報集計時は `expenses.amount_settled` を真実の源とし、`activity_logs.expense_amount` は表示用キャッシュとする
- `expenses` テーブルが多用途化するが、`expense_kind` で明示的に区別すれば混乱しない
- Day 5 で承認展開と並行して実装(設計コストはやや増えるが、要件は明確)

## ADR-015: 承認ワークフローは PoC ではクライアント状態機械として実装する

**確定日:** 2026-06-13

### Context
- ADR-012 で多段階承認(担当課 → 受入団体 → 企画課)を確定したが、PoC 時点では
  Supabase バックエンドが未接続(v5 アプリは `_app.tsx` のモックデータで動作)
- 「画面 = 仕様」原則(設計原則 1)に従い、まず動く画面で挙動を確定させたい
- 多段階の肝は「**承認者ごとに見えるものが変わる**」という視点の切り替え

### Decision
**PoC では承認ワークフローを manager アプリ内のクライアント状態機械として実装する。**

- `Approval` に `steps: ApprovalStep[]` と `currentStep`(0-based)を持たせる
  - `ApprovalStep = { approverType, approverLabel, status, comment?, decidedAt? }`
  - `approverType` ∈ `dept`(担当課)/ `host_org`(受入団体)/ `admin`(企画課)
  - `status` ∈ `waiting`(上位待ち)/ `pending`(自分の番)/ `approved` / `rejected`
- **承認者視点はヘッダーのロール切替で擬似再現**
  - PoC ではログインユーザー = 谷本室長(企画課)固定だが、デモのため
    `dept / host_org / admin` を切り替えられるトグルを置く
  - 切替で「今この承認者の番のもの」だけが承認タブに出る
- 状態遷移:
  - **承認**:`currentStep` を `approved` に → 次ステップを `waiting → pending`、
    `currentStep++`。最終ステップなら対象を最終承認しキューから除外
  - **差戻し**:`currentStep` を `rejected`(コメント必須)→ 全体を差戻しとして
    キューから除外、申請者へ再申請を促す(ADR-012 の「全段やり直し」に準拠)
- 各承認カード・詳細シートに **タイムライン UI**(`●━━○━━○`)を表示

### Consequence
- バックエンド未接続でも多段階承認の挙動・UX を完全にデモできる
- Year 1 で `/api/approvals/submit` + Postgres 関数(技術設計 §5.5)に置換する際、
  クライアント状態機械のロジックがそのまま仕様書になる
- ロール切替トグルは PoC 専用のデモ装置(本番では認証ロールで自動決定)。
  本番移行時に削除する旨をコードコメントに明記する
- `approveOne` / `rejectOne` の責務が「一覧から消す」から「ステップ前進」に変わるため、
  既存の単段ロジックは置換となる

## ADR-016: AI は差し替え可能なプロバイダ抽象にする(既定 Ollama)

**確定日:** 2026-06-13

### Context
- AI ベンダーは状況で変えたい(ローカル無料検証=Ollama、本番品質=Claude、CI/オフライン=決定論モック)
- 各機能(月報生成・相談・経費判定)が特定 SDK に密結合すると差し替えコストが高い

### Decision
- `src/lib/ai/` に **`AIProvider` インタフェース**を定義し、実装を 3 つ用意:
  - **`ollama`**(既定): `http://localhost:11434/api/chat` を叩くローカル LLM
  - **`anthropic`**: Claude API(`@anthropic-ai/sdk`)
  - **`mock`**: 外部依存ゼロの決定論的フォールバック(タスク別にそれらしい日本語を返す)
- `getAIProvider()` が **環境変数 `AI_PROVIDER`** を読んで実装を選ぶ
- 各 API Route はプロバイダの `generate({ task, messages, ... })` だけを呼ぶ(SDK 非依存)
- `mock` は CI / Ollama 非導入環境でも end-to-end を動かすための保険

### Consequence
- `.env` の 1 行(`AI_PROVIDER=ollama|anthropic|mock`)でベンダー切替
- Ollama を入れられない環境(本サンドボックス等)でも `mock` で全機能が動作
- 本番は `AI_PROVIDER=anthropic` にするだけ。プロンプトは Route 側に集約され共通
- プロバイダ追加(Gemini 等)は 1 ファイル足してファクトリに 1 行

## ADR-017: ローカル PoC のバックエンドは Next サーバ + SQLite

**確定日:** 2026-06-13

### Context
- v5 はそれまで静的エクスポート(`output: export`)のモック画面のみで、バックエンドが無かった
- 「ローカルで動くフルスタック」が必要。だが Supabase/Postgres を立てるのは PoC には重い

### Decision
- **DB: `node:sqlite`**(Node 22 内蔵)。`npm install` も native ビルドも不要、ファイル 1 個(`.data/app.db`)
- スキーマは v5 データモデル(docs/22)を移植。初回起動時に自動 migrate + seed(既存モックと同じ初期データ)
- **API: Next Route Handlers**(`/api/*`、`runtime=nodejs` / `force-dynamic`)
- フロントは初期描画でシードを表示し、マウント後に API の実データへ置換(フェッチ失敗時はシードのまま=オフライン耐性)
- 多段階承認の状態機械(ADR-012/015)は**サーバ側**(`/api/approvals/[id]/decide`)へ移動
- `next.config.mjs` は **`BUILD_STATIC=1` のときだけ静的エクスポート**。既定はサーバモード
- 本番移行: SQLite → Supabase(Postgres + RLS)、DB アクセス層と API 契約はそのまま流用

### Consequence
- `npm run dev` だけでフルスタックがローカルで動く(外部サービス不要)
- GitHub Pages 静的デプロイは API と両立しないため、フルスタックは Vercel / Node ホスト前提に変更
  (`.github/workflows/deploy.yml` は手動実行に変更)
- 隊員の経費申請 → 役場の承認キュー出現、という**アプリ横断フローが実データで動く**

## ADR-018: Year 1-2 本番スタックは Vercel + Supabase Tokyo + Bedrock Tokyo + AWS SES Tokyo

**確定日:** 2026-06-14

### Context
- データ主権調査(docs/23)で「自治体は法的義務としてのデータ国内保管を課していない、ガイドラインも努力規定」が判明
- ただし**自治体個別の調達仕様書(チェックシート 70-100 項目)**で実態としての国内保管圧力は発生
- 戦略 v3.2 の段階性(PoC → Year 1 ヒアリング → Year 2 県共同調達 → Year 3 全国展開)に対応した、説明責任を果たせる本番スタックを確定する必要があった
- 横須賀市 ChatGPT(2023)、三重県庁 Slack(2023)、宇都宮市 M365 等の前例から、「海外運営 + 国内 DC + ISMAP 登録」または「個人情報入力禁止 + 利用規約整理」のいずれかで自治体導入は通る
- 候補は 4 構成(最安 / バランス / ISMAP 完全 / 国産)、本 ADR ではバランスを選定

### Decision
**Year 1-2 の本番スタックを以下に確定する(1 名 ¥163 / 月、ARR 比 3.2%):**

| レイヤ | 選定 | 月額 | 根拠 |
|---|---|---|---|
| ホスト | Vercel Pro hnd1 Tokyo | ¥3,000 | Next.js 純正、Tokyo Edge、Year 2 で AWS App Runner に Docker 化移植可 |
| DB | Supabase Pro Tokyo + PITR add-on | ¥18,750 | Postgres 標準、Auth 統合、PITR 14日、Year 2 で RDS Tokyo へ dump/restore で移植可 |
| ストレージ | Supabase 100GB + Cloudflare R2 hybrid | ¥1,500 | ホット=Supabase(UI 統合)、コールド=R2(egress 無料、領収書アーカイブ) |
| **AI** | **AWS Bedrock Tokyo**(Sonnet 4.6 + Haiku 4.5、prompt caching ON) | ¥57,000 | **jp 地理境界完結**(東京+大阪)、Anthropic と直接契約なし、ADR-016 のプロバイダ抽象で切替可 |
| メール | AWS SES Tokyo | ¥750 | 圧倒的最安、ISMAP 登録 |
| 認証 | Supabase Auth(50k MAU 無料)+ Auth.js v5 | ¥0 | LINE は Auth.js 公式 Provider で補完、SAML 必要時に WorkOS 1 接続(¥18,750)後付け |
| 監視 | Sentry Developer(無料、PII scrub 必須)+ Cloudflare Web Analytics(無料、cookieless RUM) | ¥0 | Year 3 で New Relic Tokyo(2026/7 稼働)に拡張 |
| DNS / CDN | Cloudflare(無料 or Pro $25) | ¥0-3,750 | データ経路に乗らない設計、DDoS 無料 |
| ドメイン | ムームー .jp | ¥279 | 5 年 TCO 最安(¥14,366)、お名前.com の調整費 26% 罠回避 |
| **合計** | | **¥81,279 / 月** | |

**説明責任を果たす想定問答 7 件**(議会・情報政策課向け):
1. データは国内保管か → AWS Tokyo / Supabase Tokyo で完結、Bedrock は `jp.*` CRIS で日本国内のみ
2. 米国 SaaS リスクは → Anthropic と直接契約なし、AWS との契約のみ、AWS は ISMAP 登録・ガバクラ認定
3. 個人情報は安全か → 3 重防御(構造化フィールドで住民個人情報の混入を UI で抑制 + Row Level Security + TLS 1.3 / AES-256)
4. インシデント対応 → Sentry リアルタイム検知、Supabase 監査ログ 5 年、24h 以内に通知、SLA 99.9-99.99%
5. AI 学習に使われないか → Bedrock 契約で明示的に非使用、横須賀市 ChatGPT と同じ整理
6. ISMAP は → AWS は登録、Supabase / Vercel は未登録だが ISMAP 管理基準準拠でチェックシート対応、Year 3 で ISMAP-LIU 検討
7. 越境移転規制 → 物理 DC が日本国内のため非該当(個人情報保護委員会 Q&A の解釈通り)

**添付資料(調達時に必要):** データフロー図 / セキュリティチェックシート 70-100 項目記入版 / 個人情報取扱規程 / インシデント対応手順書 / 想定問答集 / データ削除手順書 / 先行自治体事例集(横須賀・三重・浜松・kintone 採用)

### 採用しなかった選択肢
- **Anthropic 直接 API**: 米国 DC 固定、データ主権説明で不利。開発時のみ使用、本番は Bedrock 経由
- **Azure OpenAI Japan East**: GPT-4o は安いが Regional Standard 必須で配信モデル限定、Claude 不可
- **GCP Vertex Tokyo Gemini**: Gemini は Tokyo 配信だが DRZ 契約不可、Claude は Tokyo 非対応
- **PFN PLaMo Prime(国産)**: コスト 1/4(¥14,000)で魅力的だが、月報生成・経費判定の品質ベンチマーク未検証 → **Year 2 で再評価**(PoC 結果次第で AI のみ切替)
- **AWS RDS Tokyo 単体**: Supabase の Auth / Storage / 開発速度の優位を Year 1 で捨てるのは時期尚早
- **Cloudflare Workers + D1**: D1 のリージョン不指定がデータ主権で不利
- **国産フル(さくらクラウド + さくら SendGrid + Auth0 JP)**: 月 ¥39,287 で安いが、開発速度と Bedrock 品質を取りに行く方が戦略 v3.2 に整合

### Consequence
- **AI コストが全体の 70% を占める** → Bedrock の prompt caching(30% OFF)+ Batch API(月報は月末バッチ生成で 50% OFF)併用で月額 ¥40,000-45,000 まで圧縮可能、トータル ¥65,000 程度まで下げ余地
- **Supabase / Vercel は ISMAP 未登録** → Year 1-2 はチェックシートで同等性立証、Year 3 で必要なら ISMAP-LIU(1,000-2,000 万円)取得 or AWS App Runner + RDS への移植
- **AWS 移植可能設計を最初から厳守**:`output: 'standalone'` + Dockerfile + Postgres 標準 SQL のみ使用 + プロバイダ抽象を全外部依存に適用 → Year 2 の県共同調達フェーズで Vercel + Supabase を捨てる時のスイッチングコスト最小
- **段階的投資**: Year 1 ¥30,000(軽量版)から始めて、Year 2 でフル版(本提示)¥81,279、Year 3 でスタック ③ ¥124,229
- **PLaMo Prime PoC** を Year 1 後半に並行実施し、品質許容なら AI レイヤだけ ¥57,000 → ¥14,000 に置換(年 ¥516,000 削減)

## ADR-019: LGWAN 対応は段階制 ─ Year 1-2 はインターネット系のみ、Year 3+ は OEM 提携

**確定日:** 2026-06-14

### Context
- 日本の地方自治体は専用閉域網 LGWAN(Local Government Wide Area Network、J-LIS 運営)で相互接続される
- 自治体側の三層分離(マイナンバー利用事務系 / LGWAN 接続系 / インターネット接続系)が前提で、業務 PC は基本 LGWAN 接続系
- SaaS を自治体役場から使う方法は 4 つ:
  - **A. α' モデル**(令和 6 年 10 月ガイドラインで正式化)= 中間 FW 経由でホワイトリスト SaaS にローカルブレイクアウト
  - **B. β / β' モデル** = 自治体側で業務系をインターネット接続系に移す(都道府県 3 割・政令市 4 割が採用)
  - **C. LGWAN-ASP 自前取得** = J-LIS 認定 + ホスティング事業者契約 + 専用線で LGWAN 内に SaaS を置く
  - **C'. LGWAN-ASP OEM 提携** = moconavi(レコモット)/ 両備の既存ゲートウェイに乗る
- 戦略 v3.2(年商 1-3 億・大手 SI が入れない経済性の谷)と LGWAN-ASP 自前取得は明確に矛盾(初期 400-1,000 万・月 45-115 万)
- 一方、LGWAN 完全無視も中核市以上の調達で詰む可能性

### Decision
**LGWAN 対応を 3 フェーズの段階制で進める。自前 LGWAN-ASP 取得は終生回避する。**

#### Phase 1(Year 1-2、MVP〜ヒアリング):LGWAN 対応 ¥0
- 隊員側は 100% インターネット接続系(スマホ・現場)で完結
- 役場担当課は **インターネット系専用端末 or 私物スマホ** で利用
- 追加投資 **¥0**
- ターゲット = **緩い〜標準自治体**(豊岡・神戸・西脇・新温泉等)
- ADR-018 のスタックそのままで運用

#### Phase 2(Year 2 後半〜Year 3 前半):α' モデル対応 + ISMAP-LIU 検討
- 自治体側に **中間 FW ホワイトリストへ自社 FQDN 追加** を依頼(技術的にはアプリ側の変更ほぼなし)
- **ISMAP-LIU 登録**:初期 500-1,500 万、年更新 200-500 万
  - 投資判断ライン: **ARR 2,000 万超で検討、3,000 万超で着手**
- ターゲット = 中核市・特別区・α モデル維持の厳格自治体

#### Phase 3(Year 3+、ARR 3,000 万超):LGWAN-ASP OEM 提携
- **自前 LGWAN-ASP 取得は絶対しない**(初期数千万・月数百万は単独 SaaS では回収不可)
- **moconavi(レコモット)/ 両備の LGWAN クラウドゲートウェイと OEM 提携** で対応
  - 初期 100-300 万 + 売上の 20-30% シェア
- ターゲット = 県庁本体・厳格自治体(機密性 3 データ扱う案件)

#### 設計で先回りすること(コード側で担保)
1. **固定 IP CIDR ベースのアクセス制御**(自治体ごとのホワイトリスト)
2. **`lgwan.xxx.jp` 別ドメイン分離可能性**(インターネット用 `app.xxx.jp` と分けられる構造)
3. **メール送信元抽象**(`nodemailer` 経由で AWS SES → 自治体専用メールサーバへ切替可能)
4. **Bedrock の代替経路準備**:LGWAN 内では Bedrock 直叩き不可 → ADR-016 のプロバイダ抽象で **国産 LLM(PLaMo オンプレ / NTT tsuzumi)** への切替を吸収
5. **オンプレ移植可能 Docker 構成**(既に ADR-018 で確保済み)→ OEM ゲートウェイのセンターに丸ごとデプロイ可能

### 採用しなかった選択肢
- **自前 LGWAN-ASP 即取得**: CAPEX 数千万 + OPEX 月数百万、戦略 v3.2 の経済性と矛盾。10 自治体規模でも年 1,800 万円の固定費 = 単独取得は事業崩壊リスク
- **LGWAN 完全無視**: 中核市以上の調達で詰む可能性。Phase 2 の α' モデル対応 + ISMAP-LIU はやる
- **kintone のゲートウェイに乗る**(両備 R-Cloud Proxy 経由でサイボウズと一体化): サイボウズ依存が強すぎ、戦略 v3.2 の「kintone 弱点の逆張り」に反する
- **Phase 1 から ISMAP-LIU 取得**: ARR 1,000 万未満で年 500 万固定費は無謀

### Consequence
- **Year 1-2 の追加コスト ¥0**、ADR-018 スタックのまま緩い〜標準自治体の市場を取りに行ける
- **2024-2026 の構造変化** が追い風:
  - デジタル庁が三層分離廃止 → ゼロトラスト移行方針(2024-05 河野大臣発言)
  - LGWAN 高度化(第 5 世代)+ LGCS で、**ガバクラ上の SaaS は LGWAN からシームレス**になる方向
  - 令和 7 年 4 月の少額随契緩和(委託 200 万円まで)= 随契調達可能
- **OEM ゲートウェイの存在が前提** ─ moconavi / 両備が事業継続している限り Phase 3 は実行可能。両社が撤退した場合は代替を再評価
- **設計上の先回り 5 項目**(IP 制限 / ドメイン分離 / メール抽象 / LLM 切替 / Docker)を最初から実装することで、Phase 移行コストを最小化

## 横断的な設計原則(v5 全体)

1. **画面 = 仕様**(ドキュメント先行ではなく画面試作から逆方向にドキュメントを寄せる)
2. **AI は判定しない**(視点と材料のみ。最終判断は人間に残す)
3. **業務ツール調**(白基調 + slate、色は機能で意味を持つ場合のみ)
4. **検索エンジン型 UI**(中央の検索ボックスを入り口、結果リストで深掘り)
5. **目的別の AI 相談**(全般対応より用途特化、精度を優先)
6. **構図でメリハリ**(色を増やさず階層を作る)
7. **隊員は実名 opt-in、関係者は自動匿名化**(ネットワーク形成と保護の両立)
8. **承認は多段階ルートで隊員ごとに定義**(現場運用差を吸収)
9. **お知らせ・ルール・通知は右上ベルに統合**(画面増やさず、ピン留めで常設)
10. **経費は二系統動線 + 親子構造**(現場の入力パターンの違いを吸収)
11. **AI はプロバイダ抽象で差し替え可能**(ollama/anthropic/mock、env 1 行で切替)
12. **ローカルはサーバ + SQLite で外部依存ゼロ**(本番は Supabase へ流用)
13. **本番は国内 DC + ISMAP 認定ベンダー経由で説明責任を果たす**(Vercel + Supabase Tokyo + Bedrock Tokyo + AWS SES Tokyo、AWS 移植可能設計を厳守)
14. **LGWAN 対応は段階制で OEM 提携が前提**(自前 LGWAN-ASP 取得は終生回避、Year 1-2 は ¥0、Year 3+ は moconavi / 両備等の既存ゲートウェイに乗る)
