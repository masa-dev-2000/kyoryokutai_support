# ADR v5 ─ 設計判断記録

v5「シンプル版・業務ツール調」で確定した設計判断のまとめ。
各 ADR は **Context → Decision → Consequence** の三部構成。

---

## ADR-001: 役場側を「承認 / 月報 / お知らせ」の 3 タブに固定する

**確定日:** 2026-06-11

### Context
- 役場担当は副業で協力隊を見ているケースが多く、業務工数を増やせない
- 戦略 v3.2 では役場側 3 機能を「状況確認 / 情報共有 / 申請承認」と定義していたが、
  状況確認に KPI ダッシュボード / 議会報告 / ベンチマークまで含むなど範囲が広すぎた

### Decision
**画面 = 仕様** として、v5 試作で固まった以下の 3 タブを真とする:
1. **承認**(経費 / 月次報告 / 活動相談を統合)
2. **月報**(担当隊員グリッド ・ 状態一覧)
3. **お知らせ**(textarea + 一斉送信デフォルト)

設定はフッタリンクに隠す(初回・異動時のみ)。

### Consequence
- 役場担当の操作対象が「承認 1 タップ + 月 1 回の月報チェック + 必要時のお知らせ」に圧縮された
- 議会報告書 PDF / 全国事例閲覧 / KPI ダッシュボード / ベンチマーク / 県横断ダッシュボードを
  **MVP 外に降格**(ADR-002 参照)
- ドキュメント側を画面に寄せる(`docs/11_feature_inventory.md` v3 として記録)

## ADR-002: 議会報告書 PDF・全国事例閲覧(役場)・県横断 DB を MVP 外に降格

**確定日:** 2026-06-11

### Context
- 戦略 v3.2 で「議会報告書 PDF」は役場側の最大価値とされていた
- 県庁スポンサー向けには「県横断ダッシュボード」も価値
- どちらも v5 試作で UI 露出すると画面が複雑化、ADR-001 の方針と衝突

### Decision
以下を MVP 外に降格し、Year 2 で再評価する:
- 議会報告書 PDF 自動生成(役場側)
- 全国事例閲覧 UI(役場側)
- KPI / 進捗ダッシュボード(役場側)
- 全国・近隣自治体ベンチマーク
- 県横断ダッシュボード(県庁モード)
- 承認時の常時コメント必須(ADR-003 参照)

### Consequence
- 県庁提案資料では「**次フェーズの拡張余地**」として明示
- 全国事例は隊員側に集中(`docs/11_feature_inventory.md` で Tier 2.5 に格下げ)
- 県庁価値は Year 2 で別レイヤー(`/v5/prefecture` 等)として復活させる

## ADR-003: 役場の承認コメントは「差戻し時のみ必須」

**確定日:** 2026-06-11

### Context
- 戦略 v3.2 では「**承認コメント必須化**」が豊岡 kintone の弱点逆張りとされていた
- 一方で毎回コメント必須化は役場側の負担を増やし、ADR-001 と衝突
- 「承認はワンタップ」「差戻しのみ理由を求める」が現実的なバランス

### Decision
- 承認:1 タップ即決(コメント任意)
- 差戻し(「確認」ボタン):**5 文字以上のコメント必須**

### Consequence
- 双方向コミュニケーションの趣旨は「差戻し時の理由共有」に絞られた
- 役場側の心理的ハードルが下がる
- 承認自体に AI の verdict 色を残すと「AI が判定してる」見え方になるので、
  色分けは撤廃(ADR-004 参照)

## ADR-004: AI verdict の色分けを撤廃、「AI は判定しません」を全画面で統一

**確定日:** 2026-06-11

### Context
- 隊員側メンターでは「※ AI は判定しません。視点と材料のみ提供します。」を明記
- 一方、役場側承認カードの初期実装で verdict が緑「承認」/ 琥珀「確認」と色分け
- 隊員側と役場側で原則が分裂しており、戦略の整合性が崩れていた

### Decision
- 役場側の verdict カラー(緑 / 琥珀)を **slate 系に統一**
- 承認・差戻しボタンは色ではなく**文言とアイコンの違い**で区別
- 詳細シートに「※ AI は判定しません。視点と材料のみ提供します。」を再掲

### Consequence
- AI = 第三者として「材料を提供する」ポジションが全画面で一致
- 役場側の承認判断は人間に責任が残る(行政の責任構造との整合)
- 色を増やしていないので、視覚的な情報密度も上がらない

## ADR-005: 「日報」を「活動報告」に改名、活動分類を 2 軸に分離

**確定日:** 2026-06-12

### Context
- 「日報」は 1 日 1 件の前提を持つ語感
- 現実には 1 日複数件の活動を別々に記録したいケースが多い
- 「カテゴリ」を 1 軸で持つと、隊員ごとの特化テーマ(空き家担当 / 商店街活性化担当 等)に
  対応できず、全員に空き家が出るような不自然さが生まれる

### Decision
- 「日報」→「**活動報告**」に改名(1 日 N 件可能)
- カテゴリを 2 軸に分離:
  - **活動の種類**(共通固定):会議 / 出張 / 現場訪問 / 広報 / 内勤 / イベント / 振り返り / その他
  - **活動内容**(ユーザー編集可):空き家 / 移住相談 / 町報 / ... ─ 隊員のテーマに合わせて追加・削除
- 活動時間を必須入力(0.5h 単位)
- メモ欄に 5W1H ヒントを表示

### Consequence
- 役場側の「**月報**」も同じ用語感で一貫(活動報告 → 月報)
- 月報詳細でカレンダー表示できる素材が揃う(時間・経費・件数)
- 隊員ごとに活動内容ラベルが違うため、月報生成 AI のプロンプトを
  実態に合わせやすくなる

## ADR-006: 経費を「申請(事前)」「精算(事後)」のサブタブに分離

**確定日:** 2026-06-12

### Context
- 初期実装では「経費」タブの中で申請フォームに「領収書を撮る」ボタンがあった
- だが申請(事前承認)時点では領収書はまだ存在しない
- ユーザー指摘:申請と精算は別フェーズで、領収書は精算側にあるべき

### Decision
- 経費タブを「**申請**」「**精算**」のサブタブに分離
- **申請シート**(事前):タイトル / 金額 / 用途内容 のみ(領収書なし)
- **精算シート**(事後):実支出額 / 領収書添付(必須)/ 精算メモ
- 詳細シートで「未精算」状態を出し、「精算する」CTA で精算シートへ遷移

### Consequence
- 役場側の承認モードと対応が取れる:
  - 隊員「申請」→ 役場「**活動相談 / 経費(事前)**」承認
  - 隊員「精算」→ 役場「**経費(精算)**」承認
- 用語上の整理:**活動相談 = 事前承認、経費 = 精算承認**(画面で明示)

## ADR-007: AI 相談を「目的別」に分散する

**確定日:** 2026-06-12

### Context
- ヘッダー右上の「相談」ボタンは全機能対応で、抽象的だった
- 抽象的な対話より、用途を絞ったほうが AI の応答精度が上がる
- 「経費の用途を書く」「月報の来月計画を書く」「活動メモを整える」「事例を探す」など、
  目的別に必要な視点・参考情報が違う

### Decision
- 入力ごとに**インライン相談ボタン**を配置(4 箇所):
  - 活動報告のメモ欄(`daily-write`)
  - 月報の来月計画欄(`report-plan`)
  - 経費の用途欄(`expense-purpose`)
  - 事例の検索欄(`case-find`)
- 各シートは目的別の intro / hint / mock 応答を持つ(`consultMeta`)
- 応答に「**反映して閉じる**」ボタンを置き、元入力欄に AI 提案を adopt できる
- ヘッダーの「相談」は**相談メニュー**に変更(4 目的のエントリ)

### Consequence
- AI に渡るコンテキストが明示化される(`ConsultContext` 型 union)
- 全般相談を撤廃したことで、抽象的な質問のループが起きない
- 将来 LLM の system prompt は目的別に分離して設計できる

## ADR-008: Sheet をスタック管理化(`Sheet[]`)

**確定日:** 2026-06-12

### Context
- 初期実装は `sheet: Sheet | null` で 1 階層のみ
- 月報詳細(カレンダー)→ 日付タップ → 日詳細 → 「閉じる」で**月報一覧まで戻ってしまう**
- ユーザー期待:「閉じる」で 1 階層前(=カレンダー)に戻りたい

### Decision
- `sheets: Sheet[]` のスタック構造に変更
- `pushSheet(s)` / `popSheet()` / `closeAllSheets()` の API
- スタック深度 > 1 のとき、SheetHeader の「閉じる」を「**カレンダーに戻る**」等の戻りラベルに

### Consequence
- 複数階層モーダル遷移が直感的になる
- 相談シート → メニュー → 目的別シートの遷移も同じ仕組みで動く
- 将来モバイル ネイティブ実装に移しやすい構造(モーダルスタックは iOS/Android で標準)

## ADR-009: ハブ画面を業務ツール調に統一、管理者モードを追加

**確定日:** 2026-06-12

### Context
- v5 入口(/v5)は当初「Game-style UI」のグラデ + 絵文字キャラ
- 中の業務ツール調(白基調 + slate)と完全に浮いていた
- ユーザー指摘:「v5 にあってなさすぎ」
- かつ、組織管理(隊員台帳 / 職員 / 担当割当)を扱う面が役場側 3 タブには収まらず、
  別ロールが必要

### Decision
- ハブ画面を業務ツール調に統一(白基調 + slate)
- 「プレイヤー / マスター」→「**隊員 / 役場 / 管理者**」の 3 経路に
- 機能名を実装どおりに明示(活動報告 / 月報 / 経費 / 事例 等)
- トップページ(/)の v5 ラベルも「シンプル版・業務ツール調」に変更
- 管理者画面(/v5/admin)を新規追加:
  - 隊員台帳(隊員の追加・編集・退任)
  - 職員(役場側ユーザー管理)
  - 担当割当(職員 × 隊員 のマッピング)

### Consequence
- 入口から中身まで一貫した質感(白基調 + slate + 検索エンジン型)
- 役場側の「設定」(管轄隊員チェックリスト)と管理者側の「担当割当」が役割分離
  - 役場:自分の管轄を確認・調整
  - 管理者:組織全体の割当ルールを編集
- 管轄が変わる時は管理者が更新 → 役場側に自動で反映される構造

## ADR-010: レイアウトのメリハリは色ではなく「構図・余白・太さ」で

**確定日:** 2026-06-12

### Context
- 活動報告画面で「目が滑る」「どこを見るべきか分からない」ユーザー指摘
- カラフルにすればメリハリが付くが、業務ツール調 + 「白基調 + slate」の方針と衝突

### Decision
- 色を増やさず、以下の手段でメリハリを作る:
  - **太い区切り線**(`border-b-2 border-slate-900`)で主役セクションを区切る
  - 数字を 28px 黒太字(`text-[28px] font-black`)で目立たせる
  - サブセクションは細い線(`border-b border-slate-200`)+ 小ラベルで控えめに
  - カードを撤廃しリスト中心の構造に
- 経費の使用状況サマリ(2026-06-12 修正):
  - 旧:カード 3 つ + プログレスバー
  - 新:カードなし、5 行の薄いセクションに集約

### Consequence
- 視線が「主役 → サブ」の縦動線で流れる
- 色を増やさないので情報密度は上がるが視覚ノイズは増えない
- 業務ツールとしての一貫性が保たれる

## 横断的な設計原則(v5 全体)

1. **画面 = 仕様**(ドキュメント先行ではなく画面試作から逆方向にドキュメントを寄せる)
2. **AI は判定しない**(視点と材料のみ。最終判断は人間に残す)
3. **業務ツール調**(白基調 + slate、色は機能で意味を持つ場合のみ)
4. **検索エンジン型 UI**(中央の検索ボックスを入り口、結果リストで深掘り)
5. **目的別の AI 相談**(全般対応より用途特化、精度を優先)
6. **構図でメリハリ**(色を増やさず階層を作る)
