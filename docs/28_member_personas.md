# 隊員ペルソナ & ストーリー（2026-06-29）

> 🎩 Google PM 視点。実ヒアリング（おかむら=佐用町・第3セクター業務委託 / おおより=多可町・会社所属 / 匿名#1=新温泉町・CCメール運用）を土台に、隊員の2類型を人物化する。
> 狙い：MVP の画面・権限・承認フロー・通知が「誰に何を届けるか」を、抽象論でなく1人の人間の1ヶ月で検証できるようにする。

---

## なぜ2類型に分けるか（構造の核心）

地域おこし協力隊の雇用・指揮系統は大きく2型あり、**システム上の登場人物の数が違う**。

| | A. 役場直接委託型 | B. 受け入れ団体型 |
|---|---|---|
| 任用形態 | 役場が直接雇用/委託 | 役場が任用 + 受け入れ団体（会社/第3セクター/NPO）に業務委託 |
| 日々の指揮命令 | 役場担当課 | **受け入れ団体の上司** |
| 活動費・身分の管理 | 役場担当課 | 役場担当課（団体経由のことも） |
| 報告/承認の相手 | 役場（1階層） | **役場 + 団体（2階層・責任が曖昧）** |
| 典型ペイン | 役場の無関心・放置 | 「宙に浮く」・偽装請負グレー・団体で動けない |

→ **現行システムは「隊員⇔役場」の2者しか想定していない。B型には第3の登場人物（受け入れ団体）がいる。** これが MVP の最大の設計論点（末尾）。

## ペルソナ A：役場直接委託型 ── 田中 みなと（28）

**通称「放置されがちな1人配置型」**

- 兵庫県・新温泉町、着任1年目、町に1人だけの配置
- 担当：移住定住促進。役場「企画課」が直接の窓口
- 前職：神戸の広告代理店。移住者目線を買われて採用
- スマホ世代。Notion・LINE は普通に使う
- 役場とは LINE 個人 + 月1回の対面。お知らせは**課長のCCメールでまとめて飛んでくる**（#1ヒアリング・新温泉町実態）

### 痛み（Pain）
1. **役場が無関心**。月報を出しても一度もレビューされない。「問題さえ起きなければいい」空気（おかむら証言と同型）
2. **横の繋がりゼロ**。町に同期がいない。他地域が何をしているか入ってこない
3. お知らせがCCメールに埋もれ、**重要連絡を見落とす**
4. 何が経費で通るのか毎回不安。前例が分からず役場に電話で確認

### Jobs-to-be-Done
- 「活動の証拠（月120時間）を、手間なく残して身分を守りたい」
- 「役場に"ちゃんと動いている"と伝え、たまには反応がほしい」
- 「他の町の1人配置の人がどうサバイブしているか知りたい」

### 1ヶ月のストーリー（あるべき姿）
> 5月。みなとは現場から戻る車内で、スマホに**音声で**「今日は空き家2件内見同行、移住希望者の◯◯さんと2時間」と吹き込む。AIが「内見した空き家は登録予定？」と1つ質問を返し、答えると活動記録が完成。
> 経費（ガソリン代）はレシートを撮るだけ。AIが「活動費で問題なし、旅費区分」と判定材料を添える。
> 月末、AIが今月の記録から月報の下書きを5章構成で生成。みなとは2箇所直して**「役場に提出」**。企画課の担当が承認し、初めて「移住相談の件、続報ください」と**コメントが返ってきた**。みなとは「見られている」と感じる。
> 「事例」タブで"1人配置 新温泉町"に近い他県の隊員の動きを見て、孤立感が少し和らぐ。

### この人に効く MVP 機能
音声日報 / 月報AI生成→提出 / 役場コメント（双方向化）/ 通知（CCメール代替）/ 事例検索

## ペルソナ B：受け入れ団体型 ── 大寄 けんた（34）

**通称「宙に浮く業務委託型」**（おかむら・おおよりの合成）

- 兵庫県・多可町、着任2年目。**地元の第3セクター（農産加工・直売所・レストラン）に業務委託**で所属
- 担当：特産品の販路開拓・PR。日々の指示は**会社の代表（受け入れ団体の上司）**から
- 役場側は「農林振興課（事業）」と「企画防災課（活動費）」の**2課にまたがる**
- 実態はシフトでガチガチ。社員と変わらない働き方で**偽装請負リスク**を社労士に指摘された経験

### 痛み（Pain）
1. **宙に浮く構造**。会社は守りで新提案が通らず、役場は「そっちの担当だろ」と間に入らない（おかむら証言）
2. **業務委託の壁**。空き家調査で「個人情報」を理由に役場から情報提供を断られる（ANB事例）
3. 報告が二重。会社にも役場にも説明が要るが、**誰が承認者なのか曖昧**
4. やりたい事業を団体に通せない。**「他団体ではこうしている」という説得材料が欲しい**（おかむら）
5. 任期後のキャリア不安。「会社に残れない」と2年目終盤で宣告された（おかむら）

### Jobs-to-be-Done
- 「会社で動けない時に、他地域の事例を盾に企画を通したい」
- 「自分の活動を、団体とは別に"自分の実績"として役場・全国に残したい」
- 「AIに事業構想を戦略的に壁打ちしてもらい、意味のないイベントで終わらせたくない」（おおより）
- 「契約・労務のグレーを誰かに相談したい」（→ 本MVPは外部リンク集まで）

### 1ヶ月のストーリー（あるべき姿）
> けんたは展示会出展の合間に活動を音声記録。役場の活動費申請（おおより談：申請書・予算書・請求書と書類が多くアナログ）を、システム上の経費申請で**レシート撮影+AI判定**に置き換える。過去の支払い遅延トラブルも、申請ログが残るので追える。
> 新しい体験ツアー事業を会社に提案したい。「事例」タブで似た第3セクターの成功事例を検索し、AI壁打ちに「この事業は地域のためになるか、戦略的か」を評価させる。出てきた根拠を会社代表へのプレゼンに使う。
> 月報は役場（企画防災課）に提出。だが**日々の指揮は会社**なので、「この活動は会社案件か役場案件か」がシステム上きれいに切り分かると、宙に浮かずに済む。

### この人に効く MVP 機能
事例検索（説得材料）/ AI戦略壁打ち / 経費申請（書類削減・ログ化）/ 月報AI生成
**+ 未対応：受け入れ団体という第3者をシステムがどう扱うか**

## 🔴 2類型から出てくる MVP 設計論点

1. **受け入れ団体（第3の登場人物）をどう扱うか**
   - 現行は role = 隊員/admin(役場)/super のみ。**B型の「団体上司」が居場所を持たない**。
   - 選択肢：(a) MVPはB型も「役場が唯一の承認者」で割り切る（団体は画面外） / (b) 団体を閲覧専用ロールで足す / (c) Year2送り。
   - → **MVPは(a)で割り切るのが現実的**（権限分離は隊員⇔役場の1軸に集中）。ただし「この活動の指揮元は団体/役場」のタグだけ持たせると将来拡張が楽。

2. **承認フローは1段か2段か** … A型=役場1段で十分。B型に団体承認を足すとスコープ爆発 → MVPは1段（役場）に統一。

3. **通知の主用途が型で違う** … A型=CCメール代替（お知らせ見落とし防止）/ B型=申請ステータス・支払い遅延の追跡。横断通知レイヤーは両方をカバーする必要。

4. **事例検索の価値が型で違う** … A型=孤立解消（精神的）/ B型=団体説得の実務武器。後者の方がペインが鋭い → 事例カードに「どう交渉に使えるか」の角度を持たせる余地。

5. **両型に共通の核 = 音声日報 + 月報AI生成 + 事例 + AI壁打ち**。ここはMVPの確定コア。型の差は「承認の相手」と「通知の用途」に出る。
