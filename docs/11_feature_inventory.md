## 🟢 隊員側: 3 機能(行動軸で整理)

ユーザーは**この 3 つしかやらない**。AI が裏で組み立てる。

### 1. プロジェクト(計画 + 進捗管理)

**ライフサイクル**: 計画 → 進行 → 完了(→公開で「事例」化)

含まれるサブ機能:
- 対話型プロジェクト計画(AI が「やりたいこと」から KPI・効果測定を引き出す)
- 進捗ダッシュボード(目標 vs 実績、自動集計)
- マイルストーン管理
- 関連アクション・関連事例の自動紐付け
- 完了 + 公開で「事例」として全国共有(opt-in)

技術革新: ⭐⭐ AI 対話 × 構造化データ自動抽出

### 2. 日々の動き(行動・振り返り・申請)

**1 日の流れ**: 行動記録 → 夜の振り返り → 申請(必要時)

含まれるサブ機能:
- **行動記録**: 音声 / スタンプ / テキスト / 写真(現場で短く)
- **夜の振り返り**: AI が当日のアクションを起点に深掘り対話
- **申請**: 経費申請・月次報告(AI 下書き)・休暇等の定型タスク

技術革新: ⭐⭐ 音声 × Whisper × Claude による完成度の高い記録

### 3. 学ぶ・相談する(事例 + AI 壁打ち)

**ユーザーの問い**: 「他はどうやってる?」「これでいいのかな?」

含まれるサブ機能:
- 全国事例検索(キュレーション + リアルタイムフィード)
- AI 壁打ち(戦略レビュー / 提案準備 / キャリア / 悩み)
- 似た事例のレコメンド
- 事例 → AI で「自分の地域向けに翻案」

技術革新: ⭐⭐⭐ 事例 DB × AI = JOIN との差別化の本丸

## 🟢 役場側: 3 機能(v3 / 2026-06-11 確定)

**設計思想**: 役場担当は副業で協力隊を見ている。**「役場の手をかけない」が最優先**。
画面 = 仕様。v5 試作で固まった 3 タブ構成を真とする。

### 1. 承認(デフォルトタブ)

含まれるサブ機能:
- 経費申請の承認 / 差戻し
- 月次報告の承認 / 差戻し
- 活動相談(事前承認系)の承認 / 差戻し
- **AI 判定材料の自動表示**(類似事例・引用元・ガードレール検知)
- 承認は 1 タップ即決
- **差戻し(=「確認」ボタン)時のみコメント必須**
- 未承認件数バッジ

含まれないサブ機能(MVP 外):
- 承認時の常時コメント必須化 → Year 2 で再評価
- 承認ワークフロー(多段承認・代理承認) → Year 2

### 2. 月報

含まれるサブ機能:
- 担当隊員グリッド(2-3 列 SP / 5 列 PC)
- 各カードに状態バッジ(提出済 / 承認済 / 下書き / 未着手)
- 月指定検索
- カードタップで隊員別の月報 1 件詳細(承認画面と接続)

含まれないサブ機能(MVP 外):
- 進捗ダッシュボード(プロジェクト・KPI・日報率) → 月報グリッドで代替
- 議会報告 PDF 自動生成 → **Year 2 以降**
- 全国・近隣自治体ベンチマーク → **MVP 外**
- 県横断ダッシュボード → **県庁モード**として別レイヤーで Year 2

### 3. お知らせ

含まれるサブ機能:
- textarea + 一斉送信(担当隊員全員がデフォルト送信先)
- 「変更」リンクで個別解除
- 送信履歴 + 既読率
- 送信フィードバック画面

含まれないサブ機能(MVP 外):
- 全国事例閲覧(役場側 UI) → **MVP 外**。全国事例は隊員側に集中
- 役場間の知見共有 → Year 2
- お知らせのテンプレ管理 → Year 2

### 隠し: 設定(フッタリンク)

- 管轄する隊員のチェックリスト(初回・異動時のみ)
- 通知先メール・ガードレールルール集の編集は **追加予定**(MVP 内に間に合わせる)

## 🗂️ 構造化データモデル(本質)

```
Project (ライフサイクル管理 + 事例化の核)
├─ id
├─ name, target, kpis[], effectMeasurement
├─ periodStart, periodEnd
├─ status: planning | active | completed
├─ isPublic: bool          ← 完了 + 公開 = 事例
└─ ownerId

Action (行動ログ)
├─ id, projectId?, timestamp
├─ type: voice | post | stamp | photo
├─ body, tags[], voiceUrl?, photoUrls[]

Review (夜の振り返り)
├─ id, userId, date
├─ conversation: { speaker, body }[]
├─ sourceActionIds[]
└─ generatedDailyReportId?

ApplicationRequest (申請)
├─ id, type: monthly_report | expense | leave
├─ status: draft | submitted | approved | rejected
├─ content
├─ submittedBy, approvedBy?, approverComment(必須)

GeneratedArtifact (AI 自動生成物)
├─ id
├─ type: daily_report | monthly_report | progress | case | outcome
├─ sourceProjectId? / sourceDate?
├─ content
└─ status: draft | approved | published

Announcement (お知らせ・通知)
├─ id, tenantId, title, body
├─ targets[], readBy[]
```

### 重要な気付き
**「事例」は別エンティティではなく Project の特別状態**:
- `Project.status = "completed" && Project.isPublic = true` = 事例
- データベースが極めてシンプル
- 隊員は「事例として残す」を別作業せず、**プロジェクト完了 + 公開のワンクリック**で実現
- AI は Project 単位で文脈を学習しやすい

## 💎 戦略 v3.2 / v3.3 との整合性(v3 後)

| 観点 | 整合 | 備考 |
|---|---|---|
| 非豊岡フォーカス | ✅ | 維持 |
| Hook(双方向コミュ) | ✅ | 役場「承認」差戻し時コメント必須 + 「お知らせ」既読管理で実現 |
| Moat(全国データ集約) | ✅ | 隊員 #3 学ぶ・相談する + Project の "事例化" |
| ボトムアップ | ✅ | 隊員 3 機能で完結する設計 |
| 経済性 | ✅ | 役場側を 3 タブに絞ったことで開発量も減 |
| Unfair Advantage | ✅ | AI × 構造化データで他社真似不可 |
| **価値の本体** | **AI が会話で引き出して自動で組み立てる** | フロー化は手段、AI が本体 |

### v3 で意図的に弱めた価値訴求(後で取り戻す)
- **議会報告書 PDF**: 県庁提案の最大価値だったが、画面複雑化を避けるため Year 2 で再提示
- **県横断ダッシュボード**: 県庁スポンサー用、別レイヤー(/v5/prefecture 等)で Year 2
- **役場側からの全国事例参照**: 隊員側 → 役場へ「事例を共有する」逆方向で代替できる
- 提案資料側ではこの 3 つを「次フェーズの拡張余地」として明示する
