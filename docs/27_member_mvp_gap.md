# 隊員画面 MVP ギャップ整理（2026-06-29）

## 前提：MVP 隊員側の定義

CLAUDE.md の確定スコープ。隊員側は **3 機能 + 横断レイヤー**。

1. **プロジェクト**（計画 + 進捗管理 / ライフサイクル: 計画→進行→完了 / `isPublic` で事例化）
2. **日々の動き**（行動記録 + 夜の振り返り + 申請）
3. **学ぶ・相談する**（事例検索 + AI 壁打ち）
4. 横断: **通知**

確定 MUST（オーナー判断）:
- 音声ファースト日報 + AI 質問補完
- 月次報告 AI 生成
- 全国事例検索 DB

> ⚠️ **注意**：実装は ADR-020 で「日報タブ廃止・4 タブ統合（活動 / 経費 / 連絡 / 事例）」に舵を切っており、CLAUDE.md の「3 機能（プロジェクト中心）」とは**構造が乖離している**。下記ギャップは CLAUDE.md の MUST を真とした場合の不足。実装側 ADR を真とするなら一部は「降格済み」とも解釈できるため、**どちらを MVP の正とするかをまず決める必要がある**（→ 末尾の論点参照）。

## MVP までに足りないもの（優先度順）

### 🔴 P0：MUST だが未実装・ダミー

| # | 項目 | 現状 | 必要な作業 |
|---|---|---|---|
| 1 | **音声ファースト日報** | アクティブ実装に皆無。`_archive/lab/action-log/v3-voice` 等に試作のみ | 録音 UI（MediaRecorder）→ 文字起こし API → AI 質問補完 のパイプライン新設 |
| 2 | **AI 質問補完** | consult に「整理」はあるが、記録時の能動的な深掘り質問は無し | 活動記録フローへの対話的補完の組み込み |
| 3 | **月報の提出フロー** | 「役場に提出」ボタン（`_app.tsx:1769`）は **onClick なしのデッドボタン**。`PDF 出力` も同様 | 提出 API（POST/PATCH `/api/monthly-reports`）+ status 遷移 draft→submitted の配線 |
| 4 | **月報生成結果の永続化** | `/api/ai/monthly-report` は生成のみ。`monthly-reports` は **GET のみ** で保存されない | 生成 draft の保存・再編集・取得 |
| 5 | **領収書の本番保存** | FileReader で DataURL プレビューのみ。Storage 未接続 | Supabase Storage へのアップロード配線 |

### 🟡 P1：CLAUDE.md MUST だが実装方針が未確定

| # | 項目 | 現状 | メモ |
|---|---|---|---|

## 先に決めるべき論点

1. **MVP の「正」はどちらか**：CLAUDE.md の「プロジェクト中心 3 機能」か、ADR-020 の「4 タブ（活動/経費/連絡/事例）」か。
   - 4 タブを正とするなら、項目 6（プロジェクト）は MVP 外に正式降格でき、P0 は「音声・月報提出/永続化・領収書保存」に絞れる。
   - CLAUDE.md を正とするなら、プロジェクト機能の新規実装が最大の残作業になる。
2. **音声日報の実装深度**：録音+文字起こしだけか、AI 質問補完まで含めるか。
3. **β リリースの DB**：SQLite のまま小規模 β か、Supabase 移行を先に済ませるか。
