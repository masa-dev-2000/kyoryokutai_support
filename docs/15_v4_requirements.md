# v4 要件書 v1 ─ AI が第三の当事者になる協力隊サポート

**確定日**: 2026-06-10
**根拠**: v3 ラボ(docs/13/14)+ v4 設計セッション(2026-06-10)
**位置づけ**: 「機能の引き算」と「AI の役割再定義」による PoC 実装版

## 🎯 コア仮説

> **「AI = メンター + ガードレール + 複数視点を与える第三者」として、AI を第三の当事者にする。**

v3 までは「AI = 道具(ツール)」だったが、v4 では **AI は役割を持つ参加者**。
隊員・役場と並ぶ "3 人目" として、行動・判断・合意形成のスピードを上げる。

### v4 1 行ピッチ

> **「協力隊に "AI の先輩" がつく。記録すれば月報も議会報告も勝手に整い、迷えば 3 つの視点でアドバイスがくる。役場は判断するだけ。」**

### v3 との違い

| 観点 | v3 | v4 |
|---|---|---|
| AI の位置づけ | 自動化ツール | **役割を持つ第三者** |
| 判定の主体 | AI が Stop/Go/Gray を返す | **AI は視点を提供、人間が判断** |
| メンター機能 | 補助的 | **中核機能(最重要)** |
| データ前提 | 自前データ蓄積 | **Web 公開情報 + 並行蓄積** |

---

## 👥 役割分担(v4 の核設計)

| 主体 | 役割 | やること |
|---|---|---|
| **隊員** | 行動する人 | 動く・記録する・相談する |
| **役場** | 最終判断する人 | 承認する・最終的な可否を決める |
| **AI** | 第三者 | 助言する(隊員へ)+ 判断材料を整える(役場へ)+ 複数視点を提示(両者へ) |

**原則**: AI は **判定しない**。出すのは **視点と材料**、決めるのは人間。

## 📋 機能インベントリ ─ MECE 4 カテゴリ

### A. 収集と配信 ─「やったこと」が必要な人に届く

| ID | 機能 | 隊員 | 役場 |
|---|---|---|---|
| A-1 | 音声・テキスト・写真での記録 | 入力 | ─ |
| A-2 | AI による日報整形・タグ付け・分類 | 自動 | ─ |
| A-3 | 月次報告 自動生成 | レビューのみ | 承認のみ |
| A-4 | 議会報告書 自動生成 | ─ | 承認のみ |
| A-5 | 県・国 報告書 自動入力 | ─ | 承認のみ |
| A-6 | 住民向け広報文 自動生成 | ─ | 公開判断のみ |

**1 入力 → N 出力** のパイプライン。隊員は **1 回記録するだけ**。

### B. 合意形成の高速化 ─「やりたい」と「やってほしい」をすり合わせる

| ID | 機能 | AI の役割 |
|---|---|---|
| B-1 | 経費利用前の判断補助(隊員) | ガードレール + 過去事例から判定材料を提示 |
| B-2 | 経費承認 1 クリック(役場) | AI の判定材料を見て承認するだけ |
| B-3 | 活動の事前相談(隊員) | **3 視点(役場 / 地域 / 隊員)+ スモールスタート案** を提示 |
| B-4 | 提案に対する助言(隊員側 UI のみ) | 「役場目線ではこう、地域目線ではこう」+ 修正提案 |

**重要原則**:
- B-3, B-4 の **「役場目線では嫌かもしれない」系の助言は隊員側 UI でのみ表示**(politically safe)
- AI は判定しない、視点だけ
- ガードレールは「役場の理想に近い行動を促す」誘導装置として機能(ただし強制ではない)

### C. メンター ─ v4 の最重要機能

| ID | 機能 | 説明 |
|---|---|---|
| C-1 | AI メンター(常時相談可)| 「何すればいいか分からない」を防ぐ。一次相談を AI が担う |
| C-2 | 過去隊員事例ベース回答(RAG)| Web 公開情報 + 自社蓄積データから類似事例を引いて回答 |
| C-3 | 困ったときの相談窓口(感情対応含む)| AI が一次対応、必要時は人間メンターにエスカレーション動線 |
| C-4 | 卒業後ビジョン形成 | OB/OG キャリア事例(RAG) → 「こういう人はこうなった」を提示 |
| C-5 | レコメンド(他隊員の活動・地域イベント・補助金) | 隊員のプロフィール + 活動履歴から関連情報を提案 |
| C-6 | AI メンター人格 設定可能 | 自治体・隊員ごとに口調・性格・名前をカスタマイズ |

**設計思想**:
- 隊員が動けない原因 = 「何すればいいか分からない」
- 役場の人も忙しくて対応できない
- → **AI メンターが両方を救う**

### D. 計測 ─ メタ目標の可視化

| ID | 指標 | 計測方法 |
|---|---|---|
| D-1 | 役場の介入時間 | 経費承認所要時間 / AI 自動処理率 から逆算 |
| D-2 | 隊員の活動量 | プロジェクト数・イベント数・活動種類 |
| D-3 | 隊員の活動質 | AI による活動内容スコアリング(達成度・継続性) |
| D-4 | 地域インパクト | 関係人口指標・移住相談数・地元定着率 |
| D-5 | 議会説明コスト | 報告書作成所要時間 / AI 下書き利用率 |

## 🛠 技術選定(PoC レベル)

### スタック

| レイヤ | 選定 | 理由 |
|---|---|---|
| Frontend | Next.js 16 App Router(継続)| 既存資産・型付ルート・SSR |
| Hosting | Vercel | PoC 配布即時、無料枠で足る |
| DB + Auth | **Supabase**(Postgres + Auth + Storage + RLS)| 一体型、PoC 最速、pgvector で RAG も同居可 |
| AI(テキスト)| Claude Sonnet 4.6 メイン / Haiku 4.5(タグ付け等)| 日本語品質 + コスト最適化 |
| AI(音声入力)| Claude(音声直接受け)or Whisper API | 品質比較で決定 |
| **ベクトル DB** | **Supabase pgvector** | RAG 用、別 DB を立てずに一体化 |
| **RAG オーケストレーション** | LangChain or 自前(軽量を推奨)| メンター AI の core |
| ファイル | Supabase Storage | 音声・レシート画像・Word 出力 |
| 通知 | Resend(email)| PoC は email、LINE は Phase 2 |
| エクスポート | `docx` npm | 議会報告書 Word 出力 |
| 観測 | Vercel Analytics + Supabase Logs | 最低限 |

### Phase 2 以降に回すもの

- SSO・kintone 連携
- LINE / Slack 通知
- リアルタイム機能(現状は週 1 回見るだけなので不要)

## 📚 データ収集戦略(RAG シード)

PoC 開始時点では自社データ無し。**2 方面**で対処:

### ① 起点:Web 公開情報をスクレイピング/手動収集

| 情報源 | 種類 | 用途 |
|---|---|---|
| JOIN お役立ちツール | 制度・運用ガイド | ガードレール基礎・経費規定 |
| 総務省 協力隊データ | 全国統計・制度文書 | 一般 Q&A の根拠 |
| 自治体公式サイト(協力隊紹介)| 活動事例 | 過去隊員事例 RAG |
| 個人ブログ・note | 卒業後キャリア事例 | C-4 メンター回答 |
| 自治体公開議会議事録 | 制約事例 | B-3 すり合わせ材料 |

### ② 並行蓄積:導入自治体のデータを RAG に組み込む

- 隊員の記録(A-1)が蓄積されるほど C-2 の回答品質が上がる
- 自治体側のルール集(B-1, B-3 のシード)を初期セットアップ時に取り込む
- データはすべて **匿名化 + opt-in** が前提

### データ品質の限界を許容

> 「データがないのはしょうがない、一般的なことは AI に答えさせる」

PoC では完璧を狙わず、**「Web 起点 + 並行蓄積」で徐々に質が上がる**設計とする。
