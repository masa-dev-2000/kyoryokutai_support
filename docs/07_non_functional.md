## 🎯 設計原則(5つ)

1. **Privacy by Default**: 匿名化共有は OFF。明示同意でのみ ON。
2. **Least Privilege**: 隊員は自分のデータのみ、役場は担当隊員のみ、県庁は匿名統計のみ。
3. **Auditable**: 全ての RBAC 保護操作は監査ログに残る。
4. **Japan-First**: データは日本国内保管、個人情報保護法・自治体条例準拠。
5. **Lean Ops**: MVP は最小構成で回し、Year 2 で SLA 契約化。

## 1. 法令・コンプライアンス

### 1.1 必須対応
- **個人情報保護法**(2022年改正・完全施行)
- **デジタル社会形成整備法**による自治体個人情報保護条例の統合(2023年4月〜、ほぼ全国共通化)
- **マイナンバー法**(マイナンバーは扱わない=対象外の設計にする)
- **著作権法**(AI生成物の扱い、利用規約で整理)

### 1.2 自治体との契約関連
- 自治体個人情報ファイル簿への届出協力
- 外部提供の記録保管(法第31条準拠)
- 第三者提供は原則なし(統計利用・匿名加工情報は別)

### 1.3 認証・認定(Year 2 以降、取得優先度 高→低)
- [ ] ISMAP(政府情報システム向けセキュリティ評価制度)※ 取得するかは Year 2 で判断
- [ ] プライバシーマーク
- [ ] ISO 27001 (ISMS)

**⚠️ 要確認事項(ヒアリング課題)**:
- 自治体が SaaS 導入時に要求する最低限の認証レベル
- 三層分離ネットワークとの接続可否(LGWAN 接続が必須か?)

## 2. 認証・認可(RBAC)

### 2.1 認証方式
| 段階 | 方式 |
|---|---|
| MVP | Supabase Magic Link(メールリンク) |
| Year 1 後半 | + パスワード認証併用 |
| Year 2 | + TOTP MFA(役場・県庁は必須化) |

### 2.2 Role 定義
| Role | 説明 |
|---|---|
| `member` | 協力隊員本人 |
| `municipality_staff` | 役場担当職員(協力隊所管課) |
| `municipality_admin` | 役場の担当長(複数隊員を見る) |
| `prefecture_staff` | 県庁担当(Year 2〜) |
| `super_admin` | 運営(我々) |

### 2.3 Scope × Resource マトリクス

#### MVP(Year 1)
| Resource | member | municipality_staff | municipality_admin | super_admin |
|---|---|---|---|---|
| 自分の日報 | R/W | R(担当のみ) | R(自治体内) | R |
| 他隊員の日報(同自治体) | **不可**(豊岡弱点の逆張り) | R(担当のみ) | R(自治体内) | R |
| 他自治体の日報 | 不可 | 不可 | 不可 | R |
| 月次報告 | R/W(自分) | R/W/承認(担当) | R/W/承認 | R |
| お知らせ | R(受信) | W(配信) | W(配信) | R |
| チャット | R/W(自分宛) | R/W(担当隊員と) | R/W(自治体内) | R(監査時のみ) |
| 経費申請(F13) | R/W(自分) | R/W/承認(担当) | R/W/承認 | R |

#### Year 2 追加
| Resource | member | prefecture_staff | super_admin |
|---|---|---|---|
| 匿名化統計(県横断) | R(自分でON時のみ) | R(県内集計) | R |
| 匿名化事例検索 | R(opt-in時のみ) | R | R |

### 2.4 実装方針
- Postgres の Row Level Security(RLS)を核に
- アプリ層で 2 重チェック(防御の深さ)
- 権限チェック失敗は全件監査ログへ

## 3. データ保護

### 3.1 保管場所
- **データベース**: Supabase 東京リージョン(AWS ap-northeast-1)
- **オブジェクトストレージ**: Supabase Storage 東京リージョン(音声・画像)
- **AI API**: Anthropic API(データはモデル学習に使われない設定で利用)
- **バックアップ**: 日次自動、東京リージョン内で7日間保持

### 3.2 暗号化
| 対象 | 方式 |
|---|---|
| 保管時(DB) | AES-256(Supabase 標準) |
| 保管時(Storage) | AES-256 server-side |
| 通信(クライアント⇔サーバ) | TLS 1.2 以上 |
| 通信(サーバ⇔Anthropic API) | TLS 1.2 以上 |
| 鍵管理 | Supabase KMS(MVP)→ Year 2 で KMS 外出し検討 |

### 3.3 保持・削除ポリシー
| データ | 保持期間 | 削除タイミング |
|---|---|---|
| 日報・月次報告 | 退任後 5 年(総務省 交付税措置 関連書類) | 退任 5 年後に完全削除 |
| チャット履歴 | 退任後 1 年 | 退任 1 年後 |
| お知らせ | 退任後 1 年 | 同上 |
| 監査ログ | 7 年(会計検査院準拠の上限) | 7 年後 |
| アップロード画像(経費) | 退任後 5 年 | 同上 |
| 音声ファイル | 文字起こし後 30 日 | 自動削除 |

**退任時のデータオーナーシップ**:
- 退任隊員: 自分の日報・月次報告を **CSV/PDF で一括ダウンロード可能**
- 自治体: 集計データ・月次報告(承認済)は自治体が保持

### 3.4 匿名化仕様(Year 2 の F18 核心機能)

**匿名化処理(opt-in)**:
| 要素 | 処理 |
|---|---|
| 氏名 | 完全削除 |
| メール・電話 | 完全削除 |
| 自治体名 | **地域区分に置換**(例: 「丹波篠山市」→「兵庫県中部」) |
| 具体的人名(本文中) | LLM で検出・削除 |
| 具体的施設名 | カテゴリに置換(例: 「○○レストラン」→「飲食店」) |
| 活動日 | **月単位に粗視化**(日付削除) |
| タグ | そのまま保持(検索価値あり) |
| 活動内容 | LLM で抽象化(具体固有名詞除去) |

**Opt-in の設計**:
- デフォルト OFF
- 設定画面で隊員が ON/OFF 切替可能
- ON → OFF 時は既存匿名化データから該当隊員分を削除(右忘れ権)
- 同意取得時に「匿名化後のサンプル」を提示(透明性)

## 4. 監査ログ

### 4.1 記録対象イベント
| カテゴリ | イベント |
|---|---|
| 認証 | ログイン / ログアウト / 失敗 / MFA 通過 |
| 閲覧 | 他人の日報・月次報告・チャット閲覧 |
| 編集 | 日報・月次報告の作成/編集/削除 |
| 承認 | 月次報告の承認/差戻し |
| エクスポート | CSV/PDF/Word ダウンロード |
| 権限 | Role 変更、隊員-担当の紐付け変更 |
| 配信 | お知らせ配信、対象範囲 |
| 削除 | ユーザー削除、データ完全削除 |

### 4.2 ログ項目
```json
{
  "timestamp": "ISO 8601",
  "user_id": "uuid",
  "tenant_id": "uuid",
  "role": "member|municipality_staff|...",
  "action": "view_daily_log",
  "resource_type": "DailyLog",
  "resource_id": "uuid",
  "ip_address": "hashed",
  "user_agent": "...",
  "result": "success|denied|error",
  "metadata": {}
}
```

### 4.3 アクセス
- 運営(super_admin)のみ閲覧可
- 役場・県庁に定期レポートを提供(個別操作の匿名集計)
- 本人からの開示請求には応じる(個人情報保護法準拠)

## 5. パフォーマンス要件

### 5.1 応答時間目標
| 操作 | 目標(95パーセンタイル) |
|---|---|
| ページ初期表示(モバイル 4G) | < 3 秒 |
| 日報保存 | < 1 秒 |
| 日報検索(自分の1年分) | < 2 秒 |
| 月次報告 AI 生成 | < 45 秒(ローディング表示必須) |
| お知らせプッシュ配信 | < 5 秒(100人同時) |
| チャット送信 | < 500ms |

### 5.2 同時接続
- MVP: 10〜100 ユーザー/自治体、同時接続 50 以下
- Year 2: 兵庫県全体 ≒ 100 隊員 + 50 役場担当、同時接続 150
- Year 3: 3県展開時 500 ユーザー

### 5.3 モバイル前提の配慮
- PWA 実装(オフライン下書き保存)
- 電波弱エリアでも動作(農作業現場を想定)
- 音声入力のファイルサイズ圧縮(webm/opus)

## 7. アクセシビリティ・UX

### 7.1 対応目標
- **WCAG 2.1 レベル AA**(公共利用想定)
- 文字サイズ可変(16px 基準、ユーザー設定で拡大可)
- コントラスト比 4.5:1 以上
- スクリーンリーダー対応(aria 属性)
- キーボード操作のみで全機能利用可(PC 側)

### 7.2 多言語
- MVP: 日本語のみ
- Year 2: 英語対応(外国籍の隊員対応)

### 7.3 モバイルUX 配慮(隊員は農作業・運転後が多い)
- 音声入力ファースト
- タップ領域 44px 以上
- 片手操作可能な主要動線
- 明るい屋外でも読めるコントラスト

## 9. セキュリティ

### 9.1 脆弱性対応
- 依存パッケージの自動アップデート(Dependabot)
- 月次の脆弱性スキャン
- OWASP Top 10 対策を設計時に確認

### 9.2 侵入・不正アクセス対策
- Rate limiting(ログインは 5 回 / 5 分)
- WAF(Vercel 標準)
- Supabase RLS + アプリ層 2 重チェック
- 管理者操作は MFA 必須(Year 2)

### 9.3 インシデント対応
- 個人情報漏洩時の通知フロー(法第26条準拠)
- 72時間以内に影響を受けた本人・自治体へ通知
- 個人情報保護委員会への報告
