# 地域おこし協力隊サポートシステム

## プロジェクト概要
地域おこし協力隊員の「活動記録 → 自治体報告」を AI で自動化し、
副産物として全国のナレッジを蓄積するシステム。
**提案先: 兵庫県庁**

## 戦略方針(ヒアリング後 改訂版 v2)

### プロダクトコンセプト
**「非豊岡自治体向け・協力隊統合プラットフォーム」**
豊岡kintoneの "良いとこ取り" + プライバシー設計 + AI + 県横断標準化

### 真のTAM
**兵庫県内の「非豊岡」市町村(約41市町)**
- 豊岡市は既に kintone で統合システム構築済(機能では勝てない)
- その他市町村は CC メール等のアナログ運用 → 大きな伸びしろ
- 新温泉町の実態: お知らせも CC メール運用

### 勝ち筋(ヒアリングで確定した差別化ポイント)
1. **隊員⇔役場の双方向コミュニケーション**(チャット + お知らせ配信)
2. **厳格な権限分離**(隊員は自分の情報のみ・豊岡弱点の逆張り)
3. **月次報告 AI 生成**(既存価値、差別化要素)
4. **レシート写真→経費申請**(OCR+AI、豊岡対抗)
5. **県横断ダッシュボード**(県庁のスポンサー価値)
6. **匿名化した全国事例検索**(将来価値)

### なぜ C 単独ではなく拡張したか(ヒアリング結果より)
- H1(月次報告が主ペイン)は主因ではなかった
- **真の主ペイン = 隊員と役場のコミュニケーション手段欠如**
- 豊岡kintoneという先行事例があり、単なる日報AI化では弱い

### 進め方: **ヒアリングファースト**
MVP を先に作らず、まず兵庫県内隊員1〜2名にヒアリングして仮説検証する。
動くプロトタイプは「論点を引き出すためのモック」程度にとどめる。

### ターゲット(最初の10人)
- 兵庫県内、着任1年目、1人配置の隊員
- 月次報告書に2〜3時間かかっている層

### UX 前提
- **隊員側**: スマートフォン前提(現場からその場で入力)
- **管理者側(自治体・県庁)**: PC 前提(ダッシュボード閲覧・エクスポート)

### 月次報告書のフォーマット
- 兵庫県の既存テンプレは未入手 → **一般形式**でまず作る
- 章立て例: 活動サマリ / 個別活動の詳細 / 成果物 / 来月計画 / 所感・課題

### MVP スコープ(ヒアリング後に確定)
1. 日報入力(スマホ、Markdown / 音声メモ貼付)
2. AI 整形: Claude API で自治体向け月次報告ドラフト生成
3. タグ自動付与(農業/観光/教育/移住促進 等)
4. 全国検索(匿名化)で類似事例を参照

### 非スコープ(やらない)
- SNS・チャット機能
- 自治体管理画面の高度機能(v2 で検討。MVP は閲覧のみ)
- 認証連携(SSO等)

### 指標
- 北極星指標: 週あたり日報投稿数 / 人
- ガードレール: 報告書作成時間の削減率(自己申告)

## 技術スタック(初期案)

| レイヤ | 選定 | 理由 |
|---|---|---|
| Frontend | Next.js (App Router) + Tailwind | Vercel 即デプロイ、県庁提案時にURL共有が楽 |
| Backend | Next.js Route Handlers | 最小構成 |
| DB | Supabase (Postgres) | Auth もついてくる |
| AI | Claude API (Sonnet 4.6) | 日本語要約の品質重視 |
| Auth | Supabase Magic Link | 初期は軽く |

### データモデル(初期)
- `User` (id, name, municipality, assigned_at)
- `DailyLog` (id, user_id, date, body_md, voice_url)
- `MonthlyReport` (id, user_id, year_month, draft_md, status)
- `Tag` (id, name, category)
- `DailyLogTag` (log_id, tag_id)

## 兵庫県庁提案の観点(忘れない)

- **県の価値**: 県内全隊員の活動が横断的に見える化される
- **市町村の価値**: 報告書提出が早く・質が揃う
- **隊員の価値**: 作業時間が減り、孤立感が減る(他地域の似た活動が見える)
- **提案資料は MVP 完成後に作る**(動くものを見せる方が早い)
